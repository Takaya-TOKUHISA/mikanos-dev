# MikanOS 開発記録

『ゼロからのOS自作入門』を進めながら、OSの仕組みを学習するリポジトリ
## 環境
- **OS**: Ubuntu20.04(WSL2)
- **Editor**: VS Code (Remote WSL), Okteta
- **Emulator**: QEMU
## 進捗
### 1日目(2026-03-01)，第一章
- **やったこと**
    - バイナリファイル'BOOTX64.EFI'による"Hello, world!"の表示命令の作成．
    - 'BOOT64.EFI'の配置，QEMUでの起動．
    - C環境での"Hello, world!"の表示命令の実現とQEMUでの表示．
- **学び**
    - 文字の表示には空ファイルをFAT形式でフォーマットし，そこにバイナリファイルを書き込む．
    - EDK II環境をとりいえるためにansibleを用いたが，githubのバージョン管理機能をおろそかにしており，Ubuntu20.04に対応するバージョンではなく，エラーに見舞われた．
- **考察**
    - 今までのCのコンパイルはgccで行っており，これは対象のOSに沿った形にコンパイルされる．今回は作成したOS(Ubuntu20.04)である一方実行ファイルのルールはwindows形式であるため，windows向けのバイナリファイルを作成する必要があるという部分で少し複雑に見えると考えた．
    - gccではmainから始めるが，今回はEfiMainを指定してスタートする位置を自分で決めている．
    
---
### 2日目(2026-03-02)，第二章
- **やったこと**
    - EDK IIを用いたハローワールドアプリの作成．
    - そのためのエントリポイント等の設定と，EDK IIでビルドするための設定．
    - buildを用いた機械語への翻訳．
    - メモリマップを取得するプログラムの作成．
- **学び**
    - １日目ではEDKを用いず，直接ライブラリを書き込むことで実現したが，ヘッダファイルを記述してUEFI等の機能をインクルードすることでコードを省略できた．
    - buildコマンドを用いる内部では，.dscファイルを読み込んでプログラムの全体像を把握し，
    .infファイルを読み込むことでソースコードやプロトコルなどを把握する．
    そしてそのリストに従ってコンパイラ，リンカを動かして.oや.dllに変換し，最終的にUEFIが読める.efiファイルに変換する．
    - gEfiの様々なプロトコルを活用することで，memmapというメモリの全体図を記録するCSV形式のファイルを作成し，各領域の状態を確認する方法を得た．
    - git diffやdiffのを使って自分のコードとお手本の差分を確認し修正する手法を得た．
    - コードを更新したにも関わらず，それに対応するプロトコルのguidを.infファイルに書き忘れてエラーに見舞われた．
- **考察**
    - グローバルな変数等は先頭にgをつける関数がある．
    - 今回得たメモリマップ情報は今後メモリ書き込みなどの更新の際に書き込み可能な領域を確認するために活用されると考えられる．
