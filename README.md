# MikanOS 開発記録

『ゼロからのOS自作入門』を進めながら、OSの仕組みを学習するリポジトリ
## 環境
- **OS**: Ubuntu20.04(WSL2)
- **Editor**: VS Code (Remote WSL), Okteta
- **Emulator**: QEMU
## 進捗
### 1日目(2026-03-01)，第一章
- **やったこと**
    - バイナリファイル'BOOTX64.EFI'による"Hello, world!"の表示命令の作成．
    - 'BOOT64.EFI'の配置，QEMUでの起動．
    - C環境での"Hello, world!"の表示命令の実現とQEMUでの表示．
- **学んだこと**
    - 文字の表示には空ファイルをFAT形式でフォーマットし，そこにバイナリファイルを書き込む．
    - EDK II環境をとりいえるためにansibleを用いたが，githubのバージョン管理機能をおろそかにしており，Ubuntu20.04に対応するバージョンではなく，エラーに見舞われた．
- **考察**
    - 今までのCのコンパイルはgccで行っており，これは対象のOSに沿った形にコンパイルされる．今回は作成したOS(Ubuntu20.04)である一方実行ファイルのルールはwindows形式であるため，windows向けのバイナリファイルを作成する必要があるという部分で少し複雑に見えると考えた．
    - gccではmainから始めるが，今回はEfiMainを指定してスタートする位置を自分で決めている．

### 2日目(2026-03-02)，第二章
- **やったこと**
    - EDK IIを用いたハローワールドアプリの作成．
    - そのためのエントリポイント等の設定と，EDK IIでビルドするための設定．
    - buildを用いた機械語への翻訳．
    - メモリマップを取得するプログラムの作成．
- **学んだこと**
    - １日目ではEDKを用いず，直接ライブラリを書き込むことで実現したが，ヘッダファイルを記述してUEFI等の機能をインクルードすることでコードを省略できた．
    - buildコマンドを用いる内部では，.dscファイルを読み込んでプログラムの全体像を把握し，
    .infファイルを読み込むことでソースコードやプロトコルなどを把握する．
    そしてそのリストに従ってコンパイラ，リンカを動かして.oや.dllに変換し，最終的にUEFIが読める.efiファイルに変換する．
    - gEfiの様々なプロトコルを活用することで，memmapというメモリの全体図を記録するCSV形式のファイルを作成し，各領域の状態を確認する方法を得た．
    - git diffやdiffのを使って自分のコードとお手本の差分を確認し修正する手法を得た．
    - コードを更新したにも関わらず，それに対応するプロトコルのguidを.infファイルに書き忘れてエラーに見舞われた．
- **考察**
    - グローバルな変数等は先頭にgをつける関数がある．
    - 今回得たメモリマップ情報は今後メモリ書き込みなどの更新の際に書き込み可能な領域を確認するために活用されると考えられる．

### 3日目(2026-03-03)，第三章
- **やったこと**
    - c++でカーネルを作成し，ブートローダからカーネルを呼び出した．最初の段階としては，待機時間中にbusy loopにならないようにアセンブリのhlt命令を呼び出し，待機状態を実現した．
    - カーネルはディスクから呼び出すのではなく，メモリ上から呼び出した．
    - フレームバッファを用いてピクセルを描画した．GOPを取得して全領域の値を255とすることで背景を白色にした．
    - カーネルにフレームバッファの先頭アドレスとサイズを渡し，待機命令の前にカーネル上でピクセルを適当に描画した．
    - これまで作成した全関数にEFI_STATUS statusを追記し，そして各命令ごとに返り値をstatusに格納し，エラー処理を行った．
- **学んだこと**
    - qemuのCLIが，ブートローダの処理がすべて終わる前から操作可能であるため，info registersでレジスタの状態を確認したときに想定の状態と異なっており，処理が終わっていないことに気が付くまでに時間がかかった．
    - これまでエラー処理をしてないことが気がかりだったのでやっとエラー処理が考慮されたコードをかけた．
    - CやC++では，short, int 等の型の大きさが定まっていないため，uint64_t等の定まった大きさの変数を使うことが重要である．
    - ピクセルのフォーマットにはRGB以外にもBGRやマスクなどの様々な手法があることが分かった．そしてBRGが結構使われているらしい．
- **考察**
    - 各ピクセルに逐一値を格納して描画するのはディスプレイによっては実は結構重い処理なんじゃないかと感じた．
    - EFI_FILE_INFOのメンバFileName[]は領域が定まっておらず，利用する際にこちらで適切に設定する必要がある．今回は愚直にファイル名の文字数を数えてハードコードしていたが，一文字増えたりしたら読み込めなくなる危険な手法なのではと考えた．
    - statusを作成してエラー処理を行ったが．if(EFI_ERROR(___))の中に処理を直接書き込むという手法もなしではないと思う．ただ，引数が多かったりすると異様に見づらくなる．
    - メモリ確保にはAny，Max，何もなしの3つの関数が用意されている．それぞれどこか空いている場所，指定したアドレス以下で空いている場所，指定したアドレスを確保の処理である．Anyは一般的なデータ等，カーネル本体等よりは重要ではなく，どこでもとりあえずおけたらいいよ程度のデータのための領域確保に使える．
    Maxのほうはあまり使い道が分からないが，32bit機器との通信だと大きすぎたら扱えない場合があるらしいので，それに使うらしい．

### 4日目(2026-03-04)，第四章
- **やったこと**
    - Makefileを作成することで，事前にコンパイルやリンクの条件を設定しておいて，makeコマンドを打つことで，カーネルの.elfファイルを作成できるようにした．
    - ピクセルのフォーマットを考慮した描画を行うために，ブートローダで必要な情報をカーネルに送信し，カーネルでフォーマットに合った出力形式で描画するようにした．
    - ピクセル描画のための共通部分を保持したクラスと，それを継承したRGBとBGR用のクラスをカーネルに作成し，各クラスで同名の描画関数を定義することで，関数内でのフォーマット判定を回避した．
    - 今のローダは必要な領域の計算が誤っていたため，正しい計算式を実装した．
- **学んだこと**
    - makeにはdockerファイルと似たものを感じた．
    - 最初は各関数が呼び出されるたびにピクセルのフォーマット判定という無駄な処理を挟んでいたが，
    クラスを実装して関数呼び出し以前で仕分けることで，無駄な処理を減らせており，このような処理の積み重ねが軽量化につながるのだと感じた．
    - 今までは単純にファイルサイズをもとにメモリを確保していたが，これではLOADセグメントの.bssセクションが無視されてしまうため，より詳細に先頭アドレスと終了アドレスを計算して適切なサイズを獲得することが必要だとわかった．これをしないと.bssのための物理メモリが足りなくなってしまう．
- **考察**
    - 参考書では面倒だという理由でBitMaskとうのフォーマットの実装をしていなかったので，自分でアルゴリズムを理解して処理を付け加えてみたいと考える．
    - メモリ管理の時にいっそのこと余分にとれば.bssセクション分は取れるのではないのかと考えた．
    普通のアプリ開発であれば未初期化変数の数はそれで十分な量であることが多いが，今後実装するであろうページ表など巨大な配列を未初期化変数として宣言することがあるので，それではうまくいかないらしい．
    - ページ表は実際今の64bitプロセッサでは４段，５段と段数がどんどん増えていっているようなので，綿密に計算することがメモリ管理には必要といえそう．

### 5日目(2026-03-05)，第五章
- **やったこと**
    - フォントをバイナリデータとして自作して，それをピクセル描画を用いて描画した．
    - フォントと画面描画，メイン処理と複雑になってきたので，それぞれを別のファイルに分割して保存し，それらをインクルードすることで可読性を上げた．
    - [東雲フォント](http://openlab.ring.gr.jp/efont/shinonome/)を加工して，フォントデータを組み込み，画面に描画した．
    - 一文字ずつは描画できるようになったので，sprintfを用いて文字列を作成し，それを描画するための文字列描画関数を作成した．
    - コンソールクラスを作成し，25行分の文字データを表示できるようにした．
    - これらの機能をまとめ，printfのように扱える文字列を出力するための関数printkを作成した．
- **学んだこと**
    - 文字だからと特別なことはなく，ただ，与えられた情報に沿ってピクセルを塗りつぶしているだけであるとわかった．
    - フォント一つを作るのにも結構苦労するので，漢字等の実装は外部のフォントを変換する以外ではしんどそう．
    - 一つの文字を出力する関数を，与えられた文字数分ループすることで文字列描画は実装できたが，自分がCの演習等でやってきたことと似て愚直な処理であると感じた．
    - ...のようにして引数に自由度を持たせられることを初めて知った．これを用いてフォーマットの数に依存しない文字列出力関数の実装ができた．
- **考察**
    - ピクセル描画で文字表示していることが分かったが，フォントサイズが各文字で異なると文字列ががたついたり，ほかの文字と重なったりする可能性があるのではないかと考えた．また，これに対応するためにフォントの大きさの正規化関数を実装してみても面白いかもしれない．
    - 文字列出力関数を実装したが，フォーマットと引数の数の不一致によって，未定義動作となってしまう可能性があるので気を付ける必要がある．
    - 文字列出力関数を実装できたので，デバッグなど様々なことが可能になったと感じている．

### 6日目(2026-03-06)，第六章
- **やったこと**
    - マウスカーソルの描画をするためにカーソルの形を定義した．
    - 背景を作成するために長方形の色塗りと枠線を塗るための関数群を作成した．
    - USBドライバを利用して，接続されているPCIデバイスの探索を行い，列挙できるようにした．
    - 出力する文字列にレベルをつけて，実際に出力する必要のある文字列だけを画面に出すように絞り込めるLog関数を作成した．
    - そのうち，xHCI(USB3.0)のUSBマウスを探索して，実際にその動きに追従する形で画面上のポインタを動かした．
- **学んだこと**
    - カーソルもピクセル単位で設定して描画していた．
    - PCIバスやその中のファンクション，デバイスのヘッダ等のデータを活用して特定の機器を探索することで，少しだけでもUSBなどのPCIデバイスの構造と内部での各機器の種類の判定原理を理解することができた．
    - メモリアドレス空間上にメインメモリ以外のメモリのように扱える部品も接続されていることを知った．xHCを制御するMMIOのアドレスをBAR0という領域から取り出して利用した．
    - 同じマウスであっても，UBS2.0やUBS3.0など，通信のプロトコルが異なる場合が往々にしてあるので，これらについて常に考え続けなければならないOS側の苦労を知った．
    - 今回実装した段階では実際にマウスを動かしてからマウスカーソルが動くまでの時間に遅延を感じたり，マウスの動きにカクつきを感じた．
- **考察**
    - 単なるカーソルの描画は実現できたが，以降で文字入力状態やタブ操作時など様々なカーソルの状態を表現する際の，状態切り替えの実現がなされるのかが気になる．
    - 今回の実装ではI/Oに対して待機を行わないビシーループな形のポーリングでの実装となったが，次章では割り込みを用いたマウスの処理を行うらしいが，割り込み処理を実現する際には必ず考えなければならない，同一資源に対しての排他制御をどのように実装するのか気になる．
    - プロトコルの違いはこれまでに出てきた画面描画フォーマット形式にも言えることであり，無法にプロトコルが乱立する状態でこの世にある手法全てに対応しようとすれば果てがない．
    そこで，標準規格の存在によってOS側で対応すべきプロトコルが制限され，ほとんどの機器が互いに適切に通信できているのだと考えた．
    - 実用例は挙げられなかったが今回実装したLog関数の考えかたは，非常に有用なもので，必要な情報だけに絞って表示するのが便利に感じた．
    また，実際にデバッグの際にレベルを変更して問題点を探すことができたので，これ以外の自分の開発環境でも利用することでデバッグが容易になるのではないかと考えた．
