- **やったこと**
    - 割り込みハンドラを実装して，マウスの監視をポーリングから割り込みに変更した．
    - 割り込みハンドラを高速化するために，割り込みの記録のみをハンドラで処理し，それ以降の本格的な処理は後でゆっくりと行う．
    - これを実現するためにキュー(FIFO)を実装し，時系列に沿った順に矛盾なく記録処理できる構造とした．
- **学んだこと**
    - 割り込みにも方式があり，今回は xHCI が対象であるので MSI 割り込みという比較的新しい手法を利用していることが分かった．
    - メインメモリでは，書き込みが行われただけではそれがCPUに伝わらないが，0xfe00000から，0xfe00400までの範囲はレジスタ上に配置されており，書き込みをトリガーとする処理が行えるとしった．
    - これまではメッセージがなくても常にループを回し続けていたが，今回で割り込みハンドラ，キューを実装したことで効率化を図っていることが分かった．
    - 今回は"cli"による簡単なデータ競合の阻止を行った．これによって，同時な同一資源へのアクセスを阻止し，データの整合性を維持できているとわかった．
    - "cli"だけでは完全にデータ競合が阻止できないことも分かった．
- **考察**
    - 今回の手法ではキューのメッセージを確認して，なければ"hlt"命令によって休止状態に変化させた．これによってOSが省電力になり，この間は他のタスクに資源を譲渡できるなど効率的な資源管理ができている．
    - 割り込みハンドラは予測できない入力と相性がいい一方で，その間は資源を独占するという欠点がある．そのため，今回実装したような割り込みが必要な処理とそうでない処理の判別による割り込み時間の最小がが重要であり，性能に直結すると考える．
    - 割り込み記述子テーブル(IDT)に割り込みハンドラを登録することで，マウス等の割り込み要因番号とそれに紐づく割り込みハンドラのアドレスの連想テーブルとなっており，ハードとソフトの窓口として機能しているといえるのではないかと考える．

<details><summary><big><b>8日目(2026-03-08)，第八章</b></big></summary>

- **やったこと**
    - 二日目にやったUEFI BOISで取得したメモリマップをKernelMain()に渡せるようにした．
    - UEFIが作成したスタック領域，GDT，ページテーブルをOSが管理できる領域に移動させ，適切にメモリ領域の状態を判別できるようにした．

- **学んだこと**
    - 複数言語間で同じファイルを参照する際に#ifdefをつけることで片方のみそれを利用するように制限できると知った．
    - UEFI起動時に作成するデータ構造はOS側では未使用と判定されてしまうため，ブートローダが起動して適切なタイミングで，それらの領域をOSの管理下となる領域に移動させなければならないことが分かった．
    - これまででもアセンブリを使ったコードはあったが，メモリ等の資源管理を直接OSで行おうとする際には，
    高級言語であるC++では行えず，アセンブリを用いてretfやlgdt等の命令を行わなければならないとわかった．
    - また，これらのためにはエントリポイントをアセンブラ側で定義しておき，そちらで新しいスタック領域を定義してから本来のエントリポイントに移動するという動作が必要であるとわかった．
    - 今回は実際に4段のページング構造を設定したが，やっていることは2段のページングと変わらず，多段にすることは，例えるならば完全木の高さを高くしているだけのことだと感じる．実際のページングの実装はまだらしい．
- **考察**
    - 今回はAllocateでループを用いて必要なフレーム数分連続した空き領域を探して割り当てるFirst Fitを利用した．これでは，使用領域が増えると空き領域が見つからず，時間計算量がO(N)に近づく，そのため，初めから，空き領域のみを管理するFree Fitなどの管理方法が考えられる．
    - 空き領域が連続である必要があることで，小さな数のフレームが大量にまばらに空いている際に，総空き領域は足りていても領域を確保できないことが考えられる．そのため，この解決策として，離れた領域でも仮想的に連結させて処理する方法が考えられる．また，処理の重さに目をつむると，定期的に各状態で固まるよう領域の再配置を行い，空き領域とそうでない領域の境界を定義する手法が考えられる．
    - 今回はフレーム単位で管理しているため，4KiB単位でしか領域を確保できず小さいデータが大量にメモリ確保を要求するとその領域の大部分が無駄になる．そのため，各フレームで複数のオブジェクトを管理できる仕組みが必要である．
    - 今回のAllocateだけでは確保した領域が足りなかった場合に手動で解放してからもう一度適切な領域を確保する必要があり，動的な資源を管理する際に不便である．そのため，既にある領域に追加して領域を確保するなどのすでに取得した領域への操作があると便利だと考える．

    - 前考えていたように各ピクセルへの処理が重くて，それをピクセルフォーマット形式を利用した memset で一括で処理を行うことで高速化できるとわかった．
    - また，描画処理の高速化には同様の形式のバッファを用意してこちらで全部処理してから一括で渡すことで呼び出し回数を削減し，呼び出しを繰り返すことで起きる遅延を削減できるとわかった．
- **考察**
    - 高速化のために自前でバッファをメモリ領域に用意しておくということは，その分メモリ領域を圧迫してしまうということであるので，必要なバッファの大きさと現在のPCのメモリ容量との兼ね合いを考えて実装する必要がある．
    - 今回行った高速化の手法は，大学の講義で扱っていた標準入出力の原理に近しいものを感じた．標準入出力では，システムコールが遅いため，入出力をバッファにためてから一括でシステムコールを用いるという考え方で，こちらは型変換を行ったうえでメモリへの書き込みを膨大な量繰り返すことが遅延の原因だと推定して，処理の終わった自前のバッファの存在するメモリ領域をフレームバッファに memcpy で転送している．

- **やったこと**
    - カーネルメインのコードに様々な役割の処理が点在してごちゃごちゃしているので，それぞれを関数にまとめて，その役割に近いヘッダファイル，コードに委譲した．
    - 実行時間の計測を，ポーリングを使って行っていたため，それを短期間の周期的な割込み処理とその回数のカウントによって実現した．
    - 複数の論理タイマーを優先度付きキューに記録し，それらのタイマーののタイムアウト時間を参照し，タイムアウトしていればその時刻
    を含む通知を送るようにした．
    - 現在実装している，Local APIC タイマは各PCのtick間隔に依存しており，それらは大きくばらつきがあり，1カウントの秒数が分からない．そのため，ACPI PMタイマという動作周波数が分かっているタイマを基準として，カウントの周期を測定する機能を作成するため，
    そのRSDPを取得した．
- **学んだこと**
    - やっぱり一つの関数には特定の役割と，それ以外は関数呼び出しを用いた実装にするべきであって，そうすることで，デバッグもしやすくなるとわかった．
    - 今までのタイマはポーリングで，タイマ実装時に __asm__("sti/n/thlt")から，__asm__("sti")に代わっていて，これが機能的に退化してないか不安だったが，やっぱり当たっていた．
    - ちゃんとhltを入れて，割り込みを用いたタイマにすることで，タイマの計測頻度は落ちたものの，CPUへの負荷は低下し，実用可能なレベルでの低下であるといえる．
- **考察**
    - 現在時のタイマでは，優先度付きキューによる，非常に効率的な複数タイマ管理が実現されている．タイマ処理は排他制御のいる，割り込みハンドラ内での処理であり，この部分の実行速度がそのほかの機能にも多大な影響を与える．優先度付きキューでは，残り時間が少ないタイマから順にソートされており，ソートされていない場合にはタイムアウト確認で全タイマを確認しなければいない一方で，最初にタイムアウトしていないものに出会うまで pop() し続けて，出会えば break; して終了となり，ループ回数を大幅に短縮できると考えた．
    - 優先度付きキューで効率的な管理が実装できた．実際にはほとんどないが，すべてのタイマが同時にタイムアウトしているときの実行時間はソートしていない場合と同じである．この時，その役割を持つ Tick() 関数は全タイマをループするため，実行時間はO(N)となる，
    この間は pop 等の操作を行うため割り込みが禁止されることを念頭に置く必要がある．
